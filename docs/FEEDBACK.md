## 3. ✅ สลับชื่อภายในคู่เตรียมไม่ได้ในคลิกเดียว (admin)

**อาการ:** จะสลับตำแหน่งสองคนในคู่เตรียมเดียวกัน (เช่น Mozz กับ Tao ให้สลับทีม) —
ต้องเอาออกทั้งสองคนแล้วใส่ใหม่. อยากกดสลับได้เลย.

**สิ่งที่มี:** ✎ swap แลกคนในคู่กับคนนอกได้อยู่แล้ว (PATCH `/pending-pairs/[id]`),
แต่ยังไม่มีปุ่ม "สลับข้างในคู่เดียวกัน".

**แก้:** เพิ่มปุ่ม ⇄ ที่การ์ดคู่เตรียม (admin) → สลับ team1↔team2 ของ PendingPair นั้น.
ทำได้ทั้ง client (สลับตำแหน่งแสดง) แต่ balanceTeams จัดทีมใหม่เองจาก 4 คน → ต้องเก็บลำดับที่แอดมินตั้งใจ.
ทางที่ตรง: PATCH endpoint รับ action `swapTeams` สลับ `team1Ids ↔ team2Ids`.

**หมายเหตุ:** `balanceTeams` จัดทีมอัตโนมัติจาก 4 คนเสมอ ตอน render → การสลับต้อง override
ให้ยึด team1/team2 ที่เก็บใน DB ไม่ใช่ balanceTeams. ตรวจ flow การ render คู่เตรียมก่อน.

## 4. ✅ ยกเลิกคู่ตายตัวแล้ว ระบบยังสุ่มให้คู่กันเหมือนเดิม

**อาการ:** จับคู่ซ้อมตายตัว (fixed pair) → ยกเลิก → ระบบยังจับสองคนนี้เป็นทีมเดิมบ่อย.

**Root cause:** `unpair` ล้าง `fixedPartnerId` สองฝั่งถูกต้อง — แต่**คู่เตรียม (PendingPair)
ที่จัดไว้ตอนยังเป็นคู่ตายตัว ค้างใน DB** โดยเก็บสองคนนี้ในทีมเดียวกัน. unpair ไม่แตะคู่เตรียมนั้น
→ พอ book มันสองคนก็ยังคู่กัน.

**แก้:** `unpair` ลบ PendingPair ที่มีคนใดคนหนึ่งของคู่ที่ยกเลิก → 4 คนกลับเข้าคิว จัดใหม่อิสระ
(เกมที่ลงสนามแล้วไม่แตะ). ไฟล์ `app/api/sessions/[id]/pairs/unpair/route.ts`.

**เทส:** ✅ code review (browser test ยาก — ต้อง setup คู่ตายตัว+คู่เตรียม บน prod DB, ไม่ทำระหว่าง live).

## 5. ✅ สุ่มคู่ (foursome) ซ้ำ 4 คนเดิมบ่อยมาก

**อาการ:** 4 คนเดิมเจอกันซ้ำหลายเกม.

**หลักฐาน (history 31.07.2026):**
- `New + Baz vs Note + Tiger` — เกม 5, 10, 15 (foursome เดิมเป๊ะ 3 ครั้ง)
- `Aof Thana + Aun vs Beachzy + Ploy` — เกม 2, 11, 17
- `Note + Tiger vs Ja + Bus` — เกม 19, 30
- `Mars/Mozz vs Tao/Aoff Metas` — เกม 18, 64, 80

**Root cause มี 2 ชั้น:**

1. **โทษซ้ำเบาเกิน** — `foursomeCost` ลงโทษเมื่อ >2 คนซ้ำแค่ `+300` เทียบ skill cost ×1000
   → ไร้ผล. แก้เป็นไล่ระดับ `REPEAT_PENALTY = {2: 4k, 3: 40k, 4: 150k}` คิดต่อเกมที่เคยเจอ
   (ซ้ำหลายครั้งยิ่งแพง) — ตรงกับที่ก๊วนขอ "ซ้ำ 2 คนโอเค, 3 มากสุด, 4 ไม่เอา".

2. **วงจรรีไซเคิล 4 คนเดิม (ตัวการจริงของ "เจอกันติด ๆ กัน")** — พอเกมจบ 4 คนที่เพิ่งเล่น
   คือคนว่างกลุ่มเดียว (คนอื่นเล่นอยู่/ถูกจองในคู่เตรียมหมด) → `syncPendingQueue` เห็นแค่ 4 คนนี้
   → จับกลุ่มเดิมโดยไม่มีทางเลือก → คู่เดิมวนลงสนามตลอด. โทษคะแนนช่วยไม่ได้เพราะไม่มี
   alternative ให้เลือก. หลักฐาน: เกม 11→15→20 (New+Baz vs Note+Tiger), 13→16→21
   (Punpun+Bankki vs BB+Wanlop) ซ้ำเป๊ะติดกันเป็นชุด.

**แก้ (2 จุด):**
- `REPEAT_PENALTY` ไล่ระดับตาม overlap (ข้อ 1).
- `planPendingAdditions()` ใน `lib/queue.ts`: กลุ่มที่จะจัดซ้ำเป๊ะกับเกมที่เคยเล่น →
  **ไม่ล็อกเข้าคิว** ปล่อยรอใน "คิวรอลงสนาม" จนเกมถัดไปจบ (ได้อีก 4 คน → รวม 8 → partition
  คละสองชุดเข้าด้วยกัน, ผลลัพธ์ปกติ = ซ้ำกลุ่มละ 2 คน). เงื่อนไขกันคอร์ทร้าง: hold เฉพาะตอน
  ยังมีคู่เตรียมอื่นป้อนสนามอยู่; คิวว่าง = ยอมซ้ำดีกว่าปล่อยสนามว่าง. ปุ่ม "จัดคู่เตรียมจากคิว"
  ที่แอดมินกดเอง ส่ง `force: true` → จัดเสมอ (กดแล้วต้องเห็นผล).

**เทส:** `lib/queue.test.ts` 9 ตัวคุมทั้งสองชั้น (hold/starvation-guard/force/mixing/leftover).

## 6. ✅ Feedback รอบเทส 2 (18 ก.ค.) — ยังซ้ำไปหน่อย + มือเวทปนกัน + ขอ cap 3 คู่

**อาการ:**
1. คู่ซ้ำยังเยอะไปหน่อย (ส่วนใหญ่ที่วงมา = คู่ซ้อมแข่ง Bus↔Ja, Note↔Tiger, BB↔Wanlop
   ซึ่งทีมเดียวกันตลอด "โดยตั้งใจ" + เกาะ RK ที่มีแค่ ~5 คน)
2. **มือเวทปนกัน**: RK เจอ S/S+/P บนคอร์ทเดียว — ก๊วนถือว่า "ไม่มีทางเจอกัน"
3. แอดมินเสนอเอง: จำกัดคู่เตรียมไว้ 3 คู่ ให้ระบบรอคนว่างเยอะ ๆ ก่อนจับ

**Root cause ข้อ 2:** เส้นทาง "เติมที่นั่งจากคนที่กำลังเล่น" (⏳ earmark) — เศษคิว 1–3 คน
ถูกตัดตามลำดับรอ (ไม่ใช่มือ) เลยมือคละกันเองตั้งแต่ต้น แล้วถูกจับยัดคู่ทันทีทุก sync
→ คู่ RK+S/P โผล่เต็มคิว (ดูภาพ: คู่ที่แย่เกือบทุกคู่มีคน ⏳). บวกกับระบบรีบจับทุกคน
เข้าคู่ทันที pool เลยไม่เคยโตพอให้จับมือใกล้กัน.

**แก้ (3 อย่าง, `lib/queue.ts` + `lib/matching.ts`):**
- `PENDING_QUEUE_CAP = 3` — sync เติมคิวได้สูงสุด 3 คู่ (นับรวมคู่จัดเอง). เกินนั้นคนรอใน
  "คิวรอลงสนาม" → pool โต → จับมือใกล้กันได้จริง. ปุ่ม "จัดคู่เตรียมจากคิว" ซ่อนเมื่อคิวเต็ม
  พร้อมข้อความอธิบาย.
- `worstPairCost()` + `MAX_MIX_PAIR_COST = 30` — เส้นแบ่งมือของก๊วน: คู่ที่มี pairing แย่สุด
  เกิน 30 (RK×N ขึ้นไป = 40/50, BG×S ขึ้นไป = 40) ถือว่า "ก๊วนไม่จัดแบบนี้":
  - `planPendingAdditions` hold กลุ่มมือปนไว้รอคนที่เหมาะ (เหมือน hold เกมซ้ำ) —
    ยกเว้นกลุ่มที่มีคู่ซ้อมแข่งคร่อมมือ (นั่นแอดมินจัดเอง) และยกเว้นตอนคิวว่าง (มีเกมดีกว่าคอร์ทร้าง)
  - `bestFillers` ปฏิเสธ filler ที่ทำให้เกิน 30 เด็ดขาด — ไม่มีตัวเลือกที่มือถึง = ไม่สร้างคู่ ⏳ เลย
    ปล่อยรอในคิวแทน

**หมายเหตุ:** คู่ซ้อมแข่งทีมเดียวกันทุกเกม = by design. เกาะมือโดดเดี่ยว (เช่น RK 5 คน)
ยังไงก็เจอกันบ่อยกว่ากลุ่มอื่น — ตามจำนวนคนจริง ไม่ใช่บั๊ก.

**เทส:** +6 ตัวใน `lib/queue.test.ts` (cap / hold มือปน / starvation guard / ยกเว้นคู่ซ้อมแข่ง /
bestFillers gate ×2). รวมทั้งชุด 56 ตัว.
